📄 DEV_STEPS.md — Пошаговые инструкции для Codex (эволюция проекта)

🎯 Цель

Этот документ описывает как Codex должен развивать проект, начиная с текущего состояния (CLI-хранилище без шифрования) и двигаясь к полнофункциональному UI и защищённому, шифрованному хранилищу.

Codex должен следовать этим шагам последовательно, не перепрыгивая через этапы и не усложняя раньше времени.

⸻

1. Этап 1 — Достроить CLI до полноценного инструмента

Codex должен улучшить существующий backup.py, но без ломания текущей логики.

Все новые функции добавляются в виде отдельных подкоманд.

✔ 1.1 Добавить команду verify

Функции:
	•	пройти по meta/index.json,
	•	для каждой версии проверить:
	•	существует ли объект в objects/<sha256>,
	•	совпадает ли хэш содержимого объекта с ожидаемым,
	•	вывести таблицу:

OK   /path/to/file
MISSING   /path/to/file
CORRUPTED /path/to/file


	•	вернуть:
	•	код 0, если нет ошибок,
	•	код ≠0 если есть пропавшие/битые объекты.

✔ 1.2 Добавить команду restore

Функции:
	•	аргументы: <source_path> <dest_path>
	•	найти в index.json последнюю версию <source_path>
	•	скопировать соответствующий объект из objects/<sha256> → в <dest_path>.
	•	стриминг, без больших буферов
	•	аккуратные ошибки:
	•	«файл не найден в индексе»
	•	«объект отсутствует»

✔ 1.3 Улучшить list

Сделать 2 режима:
	•	по умолчанию — краткий вывод
	•	--verbose — показывать:
	•	absolute source path
	•	все версии
	•	хэши целиком
	•	размер объекта
	•	время сохранения

✔ 1.4 Добавить stats

Показывать:
	•	сколько объектов,
	•	сколько исходных путей,
	•	общий размер objects/,
	•	самый крупный объект,
	•	средний размер,
	•	примерное дублирование (отношение версий к уникальным хэшам).

✔ 1.5 Добавить prune (сбор мусора)

Функции:
	•	найти все хэши в objects/
	•	найти все хэши в index.json
	•	удалить те, на которые нет ссылок

⸻

2. Этап 2 — Рефакторинг: выделить модули

Codex должен разделить проект на модули, чтобы CLI был просто thin wrapper:

storage_core/
    paths.py
    init.py
    index.py
    objects.py
    util.py
backup.py  → CLI

Все функции типа load_index, store_object, sha256_file, init_storage, save_index → должны находиться в storage_core/.

Принцип:
	•	CLI вызывает функции ядра,
	•	ядро не знает о CLI.

⸻

3. Этап 3 — Добавить подготовку к шифрованию (API без реализации)

Codex должен создать модуль:

storage_core/crypto.py

И добавить туда пустые функции с правильными сигнатурами:

def encrypt_stream(in_stream, out_stream, key, metadata) -> None:
    ...

def decrypt_stream(in_stream, out_stream, key, metadata) -> None:
    ...

def derive_key_from_password(password: str) -> bytes:
    ...

def generate_recovery_key() -> str:
    ...

def split_key(master_key: bytes) -> dict:
    ...

Но пока не реализовывать шифрование — только API, документацию и типы.

И Codex должен обновить index.json-структуру, добавив поля:
	•	"plain_hash" (когда появится крипта)
	•	"enc_hash"
	•	"encryption_scheme_version"

Но пока не заполнять их реальными данными.

⸻

4. Этап 4 — Подготовка к UI (общий API для будущего GUI/TUI)

Codex должен сформировать модуль:

storage_core/api.py

Где будут функции, независимые от CLI:
	•	init_storage()
	•	add_file()
	•	list_files()
	•	restore_file()
	•	verify_storage()
	•	get_stats()

UI будет вызывать только эти функции.

⸻

5. Этап 5 — Лёгкий TUI (через textual или curses)

Codex должен создать интерфейс:

tui/main.py

Который будет показывать:

Экран 1: список файлов
	•	список исходных путей
	•	выбор конкретного файла

Экран 2: информация о файле
	•	версии
	•	хэши
	•	размер
	•	кнопки:
	•	Restore
	•	Verify
	•	Show metadata

Экран 3: операции
	•	verify all
	•	stats
	•	prune

Важно:
	•	UI не должен содержать логику работы с файлами — только вызывать API в storage_core.

⸻

6. Этап 6 — Реальное шифрование

Codex должен:

✔ Реализовать crypto.encrypt_stream и crypto.decrypt_stream
	•	использовать AES-GCM,
	•	потоковое шифрование,
	•	метаданные включать в AAD,
	•	генерировать отдельный nonce для каждого объекта.

✔ Доработать add / restore
	•	при add → шифровать,
	•	при restore → дешифровать.

✔ Обновить индекс
	•	заполнять plain_hash и enc_hash.

⸻

7. Этап 7 — Touch ID / пароль учётки

Codex должен создать модуль:

storage_core/auth_mac.py

И реализовать:

def require_local_auth(action: str) -> bool:
    ...

Поведение:
	•	Если Touch ID доступен → показать диалог,
	•	Если нет → запросить пароль учётки через AuthorizationExecuteWithPrivileges/Security Framework,
	•	Если отказ → запретить операцию.

Вызовы require_local_auth() должны быть в CLI + UI при доступе к ключам и восстановлению.

⸻

8. Этап 8 — UI 2.0 (полноправный интерфейс)

После шифрования и auth:
	•	показать индикаторы шифрования,
	•	отображать ключи/метаданные,
	•	добавить “lock/unlock storage”.

⸻

📌 Важные указания Codex’у

Codex должен соблюдать:
	•	не менять существующую структуру хранилища без необходимости,
	•	не ломать обратную совместимость,
	•	все изменения — модульно,
	•	не переписывать CLI при добавлении новых модулей,
	•	шифрование внедрять только после завершения CLI и рефакторинга,
	•	UI строить поверх API из storage_core/api.py.
